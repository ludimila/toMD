# to.MD v1.1 — Design: modernização para usuários não técnicos

**Data:** 2026-07-13
**Status:** aprovado (Approach B)
**Contexto:** o to.MD é um app de desktop Windows (PySide6 + Docling) que converte
documentos para Markdown. Foi criado por Mateus Freitas Gonçalves (advogado); os
usuários finais são advogados sem conhecimento técnico. Ludimila mantém o código.
Decisões de produto já tomadas: continua sendo app de desktop (documentos jurídicos
confidenciais nunca saem da máquina do usuário); o repositório será público; a
licença proíbe uso comercial por terceiros.

## Objetivos

1. O app fica mais fácil e mais tolerante para usuários não técnicos.
2. O código fica seguro de evoluir (módulos pequenos, dependências pinadas, testes).
3. Ninguém mais precisa de uma máquina Windows para gerar o instalador (CI faz).
4. Advogados ficam sabendo de novas versões dentro do próprio app.
5. O código público não pode ser usado para ganhar dinheiro (licença noncommercial).

## Não-objetivos (fora de escopo)

- Auto-update silencioso (baixar e instalar sozinho) — adiado; o app apenas avisa e
  abre a página de download.
- Transcrição de áudio/vídeo (pipeline ASR do Docling) — os formatos de áudio/vídeo
  saem da lista de aceitos.
- Builds para macOS/Linux.
- Assinatura de código (certificado pago; SmartScreen continuará avisando).
- Interface em outros idiomas (continua pt-BR).

## 2. Reestruturação do código

`converter_gui.py` (1.090 linhas) vira o pacote `tomd/`:

| Módulo | Conteúdo (movido de converter_gui.py) |
|---|---|
| `tomd/version.py` | `__version__` — fonte única da versão (lida pelo app, CI e instalador) |
| `tomd/formats.py` | `FORMAT_GROUPS`, `ALL_EXTENSIONS`, `is_supported_file`, `build_file_dialog_filter` |
| `tomd/web.py` | `is_url`, `normalize_url`, `URL_FETCH_HEADERS`, `_detect_html_encoding`, `_html_needs_repair`, `fetch_url_source` |
| `tomd/engine.py` | lazy-load do Docling, monkeypatch de progresso, `_build_format_options`, cache do conversor + lock, `WarmupThread`, `LoaderThread`, `ConversionWorker`, `estimate_initial_duration`, `suggest_markdown_path` |
| `tomd/theme.py` | tokens de cor, fontes, `render_app_icon`, `render_doc_glyph`, `apply_card_shadow`, QSS |
| `tomd/ui.py` | `DropZone`, `MainWindow`, `LoadingWindow`, `center_on_screen` |
| `tomd/updates.py` | checagem de nova versão (seção 6) |
| `tomd/app.py` | `main()` — bootstrap (hoje no bloco `__main__`) |

- `converter_gui.py` é apagado; `to.MD.spec` passa a apontar para um
  `run.py` raiz de 2 linhas (`from tomd.app import main; main()`), que também é o
  jeito de rodar em desenvolvimento (`python run.py`).
- `pyproject.toml` com dependências pinadas (docling, PySide6, requests,
  beautifulsoup4, html5lib) e dependências de dev (pytest).
- Apagados: `converter.py`, `testar_docling.py`, `Iniciar_Conversor.vbs`.
- `FORMAT_GROUPS` perde os grupos "Áudio / Vídeo" (nunca funcionaram: o pipeline
  ASR não é configurado) e "Legendas"/"Outros" permanecem.
- Comportamento do app não muda nesta fase — é mover código, não reescrever.

## 3. UX para usuários não técnicos

**Conversão em lote.** `dropEvent` aceita N arquivos (hoje converte só o 1º).
Fluxo: ao soltar 2+ arquivos, um diálogo pede a pasta de saída uma única vez;
cada arquivo é convertido em sequência e salvo automaticamente como
`<nome-original>.md` nessa pasta (conflitos ganham sufixo ` (2)`, ` (3)`…).
O painel de progresso mostra "arquivo 2 de 5 · página 3 de 12". Ao final, um
resumo: convertidos com sucesso e falhas (com o motivo em uma linha cada).
Cancelar interrompe a fila inteira. Um arquivo só mantém o fluxo atual
(diálogo "Salvar como").

**Arrastar links do navegador.** `dragEnterEvent`/`dropEvent` passam a aceitar
URLs `http(s)` (hoje só arquivos locais); um link solto na área de drop inicia a
conversão de URL, igual ao campo de texto.

**Erros em português claro.** Mapeamento de exceções conhecidas para mensagens
humanas; o restante cai numa mensagem genérica + detalhe técnico colapsado:

| Erro detectado | Mensagem |
|---|---|
| PDF com senha (docling/pypdfium) | "Este PDF está protegido por senha. Remova a senha e tente de novo." |
| `requests.ConnectionError` / DNS | "Sem conexão com a internet — verifique sua rede e tente de novo." |
| HTTP 4xx/5xx ao baixar link | "O site não permitiu baixar esta página (erro NNN)." |
| Arquivo corrompido / formato inválido | "Não consegui ler este arquivo — ele pode estar corrompido." |
| Sem espaço/permissão ao salvar | "Não consegui salvar em <pasta> — escolha outra pasta." |

**Log em arquivo.** `logging` para `%LOCALAPPDATA%\toMD\logs\tomd.log`
(RotatingFileHandler, ~1 MB × 3): iniciação, cada conversão (fonte, duração,
sucesso/erro com traceback). O diálogo de erro menciona onde fica o log.

**Memória de pasta.** `QSettings("toMD", "toMD")` guarda a última pasta usada
(abrir arquivo e salvar) entre sessões.

## 4. Testes

`tests/` com pytest, cobrindo só lógica pura (roda sem Docling/Qt instalados —
imports pesados continuam lazy):

- `formats`: extensões simples, composta (`tar.gz` vs `gz`), caso maiúsculo,
  rejeição de áudio/vídeo.
- `web`: `is_url`/`normalize_url`; `_detect_html_encoding` (charset no header,
  no `<meta>`, UTF-8 válido, fallback windows-1252); `_html_needs_repair`
  (página normal, `</body>` prematuro, sem `</body>`).
- `engine`: `suggest_markdown_path` (arquivo local, URL com slug, URL raiz).
- `updates`: comparação de versões (`1.0` < `1.1` < `1.10`; ignora prefixo `v`).
- Erros mapeados: cada regra da tabela acima com uma exceção sintética.

## 5. Automação de build (GitHub Actions)

- **`.github/workflows/test.yml`** — em cada push/PR: Ubuntu, Python pinado,
  instala só dev-deps leves, roda pytest (~1 min).
- **`.github/workflows/release.yml`** — em tag `v*`: `windows-latest`,
  instala dependências completas, `pyinstaller to.MD.spec`,
  `iscc installer.iss /DMyAppVersion=<versão de tomd/version.py>`,
  cria o GitHub Release da tag e anexa `installer_output/to.MD_Setup.exe`.
  A versão da tag deve bater com `tomd/version.py` (o workflow falha se divergir).
- `installer.iss` passa a aceitar a versão via `/D` (mantém default local).

Fluxo de release: atualizar `version.py` → commit → `git tag v1.1` → push da tag
→ CI publica o instalador sozinho.

## 6. Aviso de nova versão

`tomd/updates.py`: no startup da janela principal, uma thread consulta
`https://api.github.com/repos/ludimila/toMD/releases/latest` (público, sem
autenticação, timeout 5 s). Se a tag for maior que `__version__`, aparece uma
barra discreta no topo da janela: "Nova versão disponível · **Baixar**" — o
botão abre a página do release no navegador. Qualquer falha (sem rede, rate
limit, JSON inesperado) é silenciosa: registra no log e não mostra nada.
Sem nag: a barra fica visível mas não bloqueia nada nem repete diálogos.

## Ordem de implementação

1. **Fase 1 — Fundação:** pacote `tomd/`, `pyproject.toml`, apagar arquivos
   mortos, testes da lógica pura. (App se comporta igual.)
2. **Fase 2 — UX:** lote, links arrastados, erros amigáveis, log, QSettings.
3. **Fase 3 — Licença + público:** LICENSE.md, README, repo público.
4. **Fase 4 — CI:** test.yml + release.yml; release v1.1 de teste.
5. **Fase 5 — Updater:** updates.py + barra de aviso (última, pois depende de
   releases públicos existirem).

Cada fase termina com o app funcionando e commitada separadamente.

## Riscos

- **Monkeypatch de progresso** depende de internals do Docling → mitigado por
  versão pinada; qualquer upgrade de docling exige re-testar o progresso.
- **Build no CI** (torch é gigante): primeira execução do release.yml pode
  precisar de ajuste de cache/disco; validar com uma tag de teste antes de
  anunciar aos usuários.
- **SmartScreen** continuará avisando (sem assinatura) — documentar no README
  o clique em "Mais informações → Executar assim mesmo".
